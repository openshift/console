# OpenShift Console 4.23 Release Notes

This release contains no runtime breaking changes. Existing plugins built for Console 4.22 remain fully
compatible without modification when loaded in Console 4.23.

## Changes to Content Security Policy (CSP)

The default `img-src` CSP directive includes `https:` in addition to `'self'` and `data:`. This allows plugins to load images from any HTTPS origin without extending the `img-src` directive.

## Changes to Console provided CSS

PatternFly CSS upgrades from 6.4.x to 6.5.x, introducing two new themes: **High contrast** and **Glass**. If your plugin includes custom CSS that overrides PatternFly styles, test your plugin with the new themes to verify that your styles render correctly.

## Changes to plugin API

- You can now use the new `console.node/sub-nav-tab` extension type. ([CONSOLE-4946])

### New extension: `console.node/sub-nav-tab`

The `console.node/sub-nav-tab` extension allows plugins to add custom sub-tabs to the Node details tab pages. Only the Configuration tab supports adding sub-tabs at this time.

#### Example usage

```jsonc
{
  "type": "console.node/sub-nav-tab",
  "properties": {
    "parentTab": "configuration",
    "page": {
      "tabId": "my-custom-tab",
      "name": "%plugin__my-plugin~My Custom Tab%",
      "priority": 60
    },
    "component": { "$codeRef": "nodePages.MyCustomTab" }
  }
}
```

The Node resource will be passed to the component as a prop:

```typescript
import type { SubPageComponentProps, NodeKind } from '@console/dynamic-plugin-sdk';

const MyCustomTab: React.FC<SubPageComponentProps<NodeKind>> = ({ obj }) => {
  // obj is the Node resource
  return <div>Custom tab content for {obj.metadata.name}</div>;
};
```

#### Tab priorities

The UI displays tabs in priority order from highest to lowest. Default built-in tab priorities for the `configuration` parent tab include:
- Storage: 70
- Machine: 50

## Changes to shared modules and API

This release introduces no changes to shared modules.

### Improved control over toast notifications

When displaying toast notifications using `useToast` hook, Console supports persisting toasts in the
notification drawer with read/unread state and management actions. Toast notifications that opt in to
drawer persistence are subject to a visible cap (default 3) with an overflow link to the notification
drawer. Existing toast consumers are unaffected — by default, toasts remain on-screen only. ([CONSOLE-5361])

`ToastOptions` supports new optional properties for better control over toast notifications:

- `persistInDrawer` (default `false`) to persist the toast in the notification drawer when set to `true`.
  Drawer-persisted toasts default to participating in the visible toast cap and overflow link.
- `skipOverflow` (default `true`) to control whether a toast participates in the visible toast cap.
  When `persistInDrawer` is `true`, defaults to `false` unless explicitly set to `true`.
  Use `persistInDrawer: true` with `skipOverflow: true` for always-visible drawer-persisted toasts
  (e.g. long-running upload progress).
- `drawerGroup` to group toast notifications into named sections in the notification drawer. When omitted,
  toasts appear in the built-in default group, displayed as "Other Alerts" (translated by Console). Custom
  `drawerGroup` values are shown as-is; plugins are responsible for translating their own group names if needed.

### Example: persisting and grouping toasts in the notification drawer

```tsx
const { addToast } = useToast();

addToast({
  title: 'Upload complete',
  variant: 'success',
  content: 'disk.img uploaded successfully.',
  persistInDrawer: true,
  drawerGroup: 'Uploads',
});
```
